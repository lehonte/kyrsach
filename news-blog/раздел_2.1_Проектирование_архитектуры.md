2 Проектирование и разработка программного средства

2.1 Проектирование архитектуры и описание состояний программного средства

Для разработки многопользовательского веб-приложения новостного блога было принято решение использовать архитектурный паттерн Model–View–Controller (MVC). Это шаблон (паттерн) программирования, разделяющий архитектуру приложения на три модуля: модель (Model), представление (View), контроллер (Controller). Простыми словами, он позволяет изменять каждый компонент независимо друг от друга для простой разработки и поддержки веб-приложений. Такой подход делает систему более понятной: модели отвечают за данные и их структуру, контроллеры обрабатывают запросы пользователей, а представления обеспечивают отображение информации на стороне клиента.

Использование MVC также упрощает взаимодействие между компонентами. Например, когда пользователь создаёт новую статью или ставит лайк, контроллер получает запрос, передаёт его в сервисный слой, где выполняется основная логика работы — проверка прав доступа, валидация данных, сохранение статьи или обновление количества лайков. Данные при этом хранятся в моделях, которые связаны с таблицами базы данных. Такой подход делает проект более организованным и удобным для тестирования отдельных частей программы.

В административной части используется аналогичная структура. Администратор может входить в систему, управлять статьями, добавлять или изменять категории, просматривать статистику и назначать других пользователей администраторами. Все эти действия проходят через контроллеры и сервисы, что обеспечивает единый порядок обработки запросов и минимизирует ошибки.

Рисунок 2.1 – Структура проекта в среде разработки IntelliJ IDEA

Пакет config содержит настройки приложения. Здесь находится SecurityConfig, который определяет, какие страницы доступны всем, а какие — только авторизованным пользователям или администраторам. Он также настраивает JWT-аутентификацию, CORS-политику и шифрование паролей. DataInitializer проверяет наличие администратора по умолчанию в базе данных и создаёт его при первом запуске приложения.

Пакет controller включает контроллеры — классы, которые принимают запросы от пользователя. AuthController отвечает за страницы входа и регистрации, генерирует JWT-токены для авторизованных пользователей. ArticleController показывает список статей, их детальное описание, позволяет создавать, редактировать и удалять статьи, а также ставить лайки. CategoryController управляет категориями статей. AdminController предоставляет административные функции: просмотр статистики, создание категорий и назначение администраторов. UserController позволяет администраторам управлять пользователями.

Пакет dto содержит специальные классы для передачи данных между экраном и сервером. LoginRequest принимает логин и пароль при входе в систему. LoginResponse возвращает JWT-токен, имя пользователя и его роль после успешной авторизации. RegisterRequest передаёт данные при регистрации: имя пользователя, email и пароль.

Пакет model содержит основные сущности — объекты, которые хранятся в базе данных. User описывает пользователя: имя пользователя, email, пароль и роль (ROLE_USER или ROLE_ADMIN). Article описывает статью: заголовок, содержание, автор, категория, URL изображения и список пользователей, поставивших лайк. Category описывает категорию статей: название категории.

Пакет repository отвечает за работу с базой данных. UserRepository ищет пользователей по имени или email и проверяет, существует ли уже такой пользователь. ArticleRepository позволяет находить статьи по категории и получать все статьи. CategoryRepository помогает управлять категориями статей.

Пакет service содержит бизнес-логику — то есть правила, по которым работает приложение. UserService занимается регистрацией, поиском пользователей, сменой ролей и созданием администратора по умолчанию. Он также используется для авторизации. ArticleService отвечает за создание, изменение и удаление статей, проверяет права доступа (только автор может редактировать свою статью, администратор может удалять любые), управляет лайками и фильтрацией статей по категориям. CategoryService управляет категориями: создаёт новые категории и получает список всех категорий.

Пакет security содержит компоненты безопасности. JwtTokenProvider генерирует и проверяет JWT-токены для аутентификации пользователей. JwtAuthFilter перехватывает HTTP-запросы и проверяет наличие валидного токена в заголовках запроса.

Класс NewsBlogApplication — это точка входа в приложение. Он запускает Spring Boot и поднимает всю систему.

Такое разделение делает систему удобнее: каждый модуль можно проверять отдельно, и если нужно заменить какую-то часть, например базу данных, то не придется менять всю логику приложения.

После анализа требований и выбора архитектуры MVC можно перейти к описанию того, как пользователи работают с системой. Для наглядности была создана диаграмма вариантов использования, которая показывает всех внешних участников и основные действия, доступные им в приложении. Она также разделяет уровни доступа согласно ролевой модели. Диаграмма приведена на рисунке 2.2.

Рисунок 2.2 – Диаграмма вариантов использования веб-приложения

На диаграмме показаны два основных актёра системы:

1. Пользователь — обычный человек, который может зарегистрироваться, войти в систему, просматривать список статей, читать детальное описание статей, создавать свои статьи, редактировать и удалять свои статьи, ставить лайки статьям и просматривать понравившиеся статьи.

2. Администратор — это более привилегированный пользователь. Он имеет все возможности обычного пользователя, но дополнительно может удалять любые статьи (не только свои), управлять категориями, просматривать статистику системы и назначать других пользователей администраторами.

Основные сценарии для пользователя: регистрация, вход в систему, просмотр списка статей, просмотр детального описания статьи, создание новой статьи, редактирование своих статей, удаление своих статей, постановка лайков и просмотр понравившихся статей.

Основные сценарии для администратора: все, что может пользователь, плюс удаление любых статей, управление категориями (создание новых), просмотр статистики (количество статей, пользователей, категорий) и назначение новых администраторов.

Диаграмма показывает, что система разделяет права доступа. Обычный пользователь может выполнять только действия, связанные с просмотром, созданием и редактированием собственных статей. Администратор же управляет всей информацией о статьях, категориях и другими пользователями.

Диаграмма состояний показывает, как система переходит из одного состояния в другое. Она представлена на рисунке 2.3.

Рисунок 2.3 – Диаграмма состояний

Диаграмма описывает процесс взаимодействия пользователя с системой новостного блога. Пользователь начинает с экрана входа или регистрации и после авторизации попадает на главную страницу, где отображаются доступные статьи и его собственные статьи. Отсюда он может перейти к подробному описанию выбранной статьи, нажав на неё, а затем — к редактированию статьи, если он является её автором, нажав кнопку «редактировать». Также на главной странице доступна возможность создать новую статью через кнопку «создать статью», что приводит к экрану создания статьи, где пользователь заполняет форму с заголовком, содержанием, категорией и изображением. После сохранения статья отображается в списке.

Если пользователь является администратором, то с главной страницы ему также доступны переходы на страницу администратора, где он может просматривать статистику, управлять категориями и назначать администраторов. Диаграмма выражает простую и понятную структуру переходов. Она показывает, что система имеет логичную последовательность экранов.
